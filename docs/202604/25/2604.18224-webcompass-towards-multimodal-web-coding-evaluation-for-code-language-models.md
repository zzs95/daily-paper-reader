### 1. 核心问题与整体含义（研究动机和背景）
*   **核心问题**：现有的 Web 编程评估基准（如 HumanEval 或 SWE-Bench）主要关注算法正确性或后端逻辑，而忽视了 Web 开发中至关重要的**视觉保真度（Visual Fidelity）**、**交互质量（Interaction Quality）**以及**代码库级别的推理能力**。
*   **研究背景**：大语言模型（LLM）正从被动的代码助手转变为能够处理端到端 Web 工程的智能体。然而，真实的 Web 开发是一个“生成-编辑-修复”的迭代循环，现有的基准测试往往只关注单一的文本到代码生成，缺乏对多模态输入（图像、视频）和复杂生命周期任务的全面覆盖。

### 2. 方法论：核心思想与关键技术
WebCompass 提出了一个统一的多模态 Web 工程评估框架，其核心思想包括：
*   **全生命周期任务覆盖**：涵盖 3 种模态（文本、图像、视频）和 3 种任务类型（生成、编辑、修复），细分为 7 个任务类别。
*   **任务感知评估范式**：
    *   **Agent-as-a-Judge (针对生成任务)**：引入 Claude Code 作为编排者，通过 **Model Context Protocol (MCP)** 控制真实浏览器。它能动态合成 JavaScript 测试用例，模拟用户点击、滚动等交互，并结合截图进行视觉审计。
    *   **LLM-as-a-Judge (针对编辑与修复)**：采用基于检查点（Checklist）的评分机制，利用多模态大模型（如 Claude-4.5-Opus）对比修改前后的代码、日志和截图。
*   **确定性任务构建**：
    *   **生成任务**：将模糊需求转化为结构化的设计文档。
    *   **修复任务**：采用“逆向合成”法，在高质量原型中注入 11 类常见前端缺陷（如遮挡、溢出、交互丢失），并提供精确的 search/replace 标注作为金标准。

### 6. 主要结论与发现
*   **闭源模型领先**：Claude-4.5-Opus 和 Gemini-3-Pro 表现最强，且在生成、编辑、修复三项任务中表现均衡。
*   **开源差距显著**：最强的开源模型（Qwen3-VL-235B）与顶尖闭源模型之间存在约 26 分的巨大差距。
*   **视觉是瓶颈**：审美和视觉保真度（Aesthetics）是所有模型的共同短板，尤其是开源模型。
*   **任务难度差异**：编辑任务对开源模型极具挑战；修复任务中，语义缺陷（Semantic Error）比视觉布局缺陷更难修复。
*   **框架影响**：模型在 Vanilla (原生 JS) 下表现最好，Vue 框架由于单文件组件的复杂性，表现通常最差。
*   **思维链（Thinking）的双刃剑**：在生成任务中，长推理链有助于结构正确性，但可能导致“注意力稀释”，从而遗漏某些功能细节。

### 7. 优点与亮点
*   **多模态闭环**：首次将视频模态引入 Web 编程评估，捕捉动态交互。
*   **Agent-as-a-Judge 创新**：通过 MCP 协议让 AI 裁判具备“动手测试”的能力，比单纯的截图对比更接近人类 QA 流程。
*   **生态代表性**：缺陷类型和编辑任务均源于对真实社区（如 V0, GitHub Issues）的调研，具有极高的工程实践价值。

### 8. 不足与局限
*   **后端缺失**：目前仅关注前端（HTML/CSS/JS），未涉及数据库、API 开发等全栈能力。
*   **评估成本高昂**：依赖高阶模型（如 Claude-Opus）作为裁判，大规模推广的经济成本较高。
*   **动态性限制**：对于高度动态的网页（如 Web 游戏），目前的 Agent 裁判在处理实时状态转换和精确时序逻辑方面仍有挑战。
*   **数据污染风险**：作为静态基准，未来模型可能会在训练中包含类似数据，需定期更新以保持有效性。
