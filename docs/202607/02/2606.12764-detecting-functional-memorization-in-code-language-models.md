## 论文详细总结（自动生成）

这篇论文《Detecting Functional Memorization in Code Language Models》深入探讨了代码大语言模型（Code LLMs）中一种被忽视的风险：**功能性记忆（Functional Memorization）**。以下是对该论文的结构化总结：

### 1. 核心问题与整体含义
*   **研究动机**：传统的模型记忆研究主要关注“字面记忆”（Verbatim Memorization），即通过 BLEU 或编辑距离等指标衡量生成代码与训练代码在 Token 级别的重合度。然而，代码具有特殊性——同一功能可以用完全不同的语法结构实现（如重命名变量、重构循环、更改算法实现）。
*   **核心问题**：如果模型在不重复原代码文本的情况下，复现了训练集中的核心逻辑（如私有定价公式、加密逻辑或特定业务规则），现有的审计手段将失效。本研究旨在定义、检测并量化这种“功能性记忆”。

### 2. 方法论
*   **核心思想：反事实设置（Counterfactual Setup）**：为了区分模型是“记住了特定样本”还是“基于通用知识的泛化”，作者利用了 Olmo-3-32B 模型的训练透明度，对比了两个 Checkpoint：
    1.  **目标模型 ($M_T$)**：经历了中期训练（Midtraining），接触过目标代码数据。
    2.  **参考模型 ($M_R$)**：仅经过预训练，未接触过目标代码数据。
*   **检测流程**：
    1.  **提示词构造**：使用函数签名（Signature）和上下文（最多 250 Token）作为 Prompt。
    2.  **贪婪解码**：生成代码补全结果。
    3.  **多维度相似度评估**：
        *   **文本维度**：BLEU、编辑距离、最长公共子串（LCS）。
        *   **结构维度**：CodeBLEU（包含 AST 和数据流分析）、TSED（AST 树编辑距离）。
        *   **语义维度（LLM-as-a-judge）**：使用 LLaMA-3.1-70B 作为裁判，评估逻辑等价性。
        *   **执行维度**：使用 HyClone 框架，通过生成测试用例并运行代码，对比输出结果的一致性。

### 3. 实验设计
*   **数据集**：从 Olmo-3 的中期训练集 **CraneCode**（Python 子集）中采样。
*   **数据筛选**：使用 LLaMA-3.1-7B 过滤掉简单的 LeetCode 题或教科书代码，保留了 **7,422 个** 具有实际业务逻辑或复杂功能的函数，并标注了泄露严重程度（Severity）。
*   **对比基准**：以参考模型 $M_R$ 的生成表现作为 Baseline。如果 $M_T$ 生成了功能一致的代码而 $M_R$ 没有，则判定为反事实记忆。

### 4. 资源与算力
*   **算力说明**：论文未详细列出训练 Olmo 模型本身的算力（因为是直接使用开源 Checkpoint），但在推理评估阶段，使用了 **LLaMA-3.1-70B-Instruct** 作为主要的语义裁判和测试用例生成器。
*   **执行环境**：为了运行 HyClone 进行执行验证，作者构建了一个标准的 Python 执行环境，并尝试通过 LLM 生成缺失的导入语句（Preamble）以提高代码可运行性。

### 5. 实验数量与充分性
*   **实验规模**：针对 7,422 个函数进行了全量测试，每组函数均在两个模型上运行并经过 10 余种指标评估。
*   **充分性**：实验设计非常严谨。通过对比 $M_T$ 和 $M_R$ 的 $\Delta$（差值），有效地排除了模型泛化能力的干扰。此外，作者还分析了函数长度、签名长度、上下文长度对记忆的影响，实验维度全面。

### 6. 主要结论与发现
*   **功能性记忆显著存在**：在文本重叠度较低（BLEU < 0.75）的样本中，约有 **3.9%** 的函数被 LLM 裁判判定为功能性记忆。
*   **执行验证的证据**：约 **0.28%** 的生成结果通过了最严格的执行一致性测试（HyClone），证明模型能够精确复现复杂的硬编码逻辑（如特定城市的佣金率表）。
*   **指标失联**：文本相似度指标与功能相似度指标的相关性较低（$r \approx 0.5$），这意味着仅靠文本去重（Deduplication）无法完全消除逻辑泄露风险。
*   **严重性无关**：模型对代码的记忆并不区分其重要程度，高敏感逻辑和普通逻辑被记忆的概率相近。

### 7. 优点
*   **定义了新维度**：填补了代码模型安全审计中“逻辑泄露”检测的空白。
*   **反事实框架的科学性**：利用开源模型的不同训练阶段进行对比，提供了比单纯搜索训练集更可靠的记忆证据。
*   **多模态评估**：结合了静态代码分析、动态执行验证和 LLM 语义理解，评估体系非常完备。

### 8. 不足与局限
*   **执行覆盖率受限**：由于环境依赖问题（如缺失第三方库、类成员变量 self 无法初始化），仅有约 **15%** 的候选函数能成功执行，这可能导致对功能性记忆的低估。
*   **LLM 裁判的偏差**：实验显示不同的 Prompt 对 LLM 裁判的评分影响显著，虽然使用了 70B 模型，但仍存在一定的主观性和不确定性。
*   **语言单一性**：目前仅针对 Python 进行了深入研究，其他编程语言（如 C++、Java）的特性可能带来不同的记忆表现。
