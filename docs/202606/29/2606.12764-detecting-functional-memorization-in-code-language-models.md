### 论文总结：检测代码语言模型中的功能性记忆 (Detecting Functional Memorization in Code Language Models)

#### 1. 论文的核心问题与整体含义
*   **研究动机**：随着大语言模型（LLM）在代码生成领域的广泛应用，关于训练数据泄露（如版权代码、私有逻辑、API 密钥等）的审计变得至关重要。
*   **核心痛点**：现有的记忆检测方法主要依赖**文本重叠**（如 BLEU、编辑距离），即检查模型输出是否与训练数据逐字匹配。然而，代码具有“功能等效性”——即使变量名、注释或控制流发生变化，其核心逻辑可能依然是训练数据的复现。
*   **核心概念**：论文提出了**功能性记忆 (Functional Memorization)**，即模型内化了训练数据的逻辑，并能以不同的代码形式（Surface Form）将其复现，而传统的文本指标无法检测到这种泄露。

#### 2. 方法论：核心思想与技术细节
论文构建了一个**反事实实验框架 (Counterfactual Setup)**，旨在区分“记忆”与“泛化”：
*   **模型对比**：
    *   **目标模型 ($M_T$)**：经过特定中期训练（Midtraining）数据暴露的模型（Olmo-3-32B 阶段性 Checkpoint）。
    *   **参考模型 ($M_R$)**：未接触过该特定数据的预训练模型。
*   **判定逻辑**：如果 $M_T$ 生成了与训练数据功能等效的代码，而 $M_R$ 没有，且 $M_T$ 的输出与训练数据文本相似度较低，则判定为“功能性记忆”。
*   **多维相似度度量**：
    1.  **文本指标**：BLEU、编辑距离、最长公共子串 (LCS)。
    2.  **结构指标**：CodeBLEU（结合 AST 和数据流）、TSED（AST 树编辑距离）。
    3.  **LLM-as-a-judge**：使用 Llama-3.1-70B 作为裁判，通过三种不同的 Prompt 评估逻辑等效性。
    4.  **执行检测 (HyClone)**：生成测试用例，在沙盒中运行生成代码与原始代码，对比输出结果。

#### 3. 实验设计
*   **数据集**：从 Olmo-3 的中期训练集 **CraneCode**（Python 子集）中采样。
*   **样本筛选**：利用 LLM 过滤掉简单的模板代码，保留了 **7,422 个** 具有实际功能逻辑（如业务规则、算法、API 调用模式）的函数。
*   **Benchmark**：以 CraneCode 中的原始函数作为 Ground Truth。
*   **对比方法**：对比了文本匹配方法与功能性检测方法在识别泄露样本数量上的差异。

#### 4. 资源与算力
*   **模型规模**：主要实验基于 **Olmo-3-32B**。
*   **辅助模型**：使用 **Llama-3.1-70B-Instruct** 作为 Judge 模型，**Llama-3.1-7B-Instruct** 用于数据清洗和严重程度标注。
*   **算力说明**：论文未明确列出具体的 GPU 型号、数量或总训练时长，但考虑到 Olmo-3-32B 的推理和 Llama-3.1-70B 的大规模调用，该实验需要相当规模的计算集群支持。

#### 5. 实验数量与充分性
*   **实验规模**：对 7,422 个函数进行了补全生成，并对每个生成结果运行了超过 10 种不同的相似度指标。
*   **分析维度**：
    *   **消融/相关性分析**：分析了文本指标与功能指标的相关性（发现相关性较低，证明了功能检测的必要性）。
    *   **敏感度分析**：探讨了函数长度、Prompt 长度、逻辑严重程度（Severity）对记忆的影响。
*   **客观性**：通过“反事实”设置（Target vs. Reference）有效地排除了模型仅靠通用编程知识“猜对”逻辑的可能性，实验设计严谨且客观。

#### 6. 论文的主要结论与发现
*   **功能性记忆确实存在**：约 **3.9%** 的样本在文本相似度极低的情况下，被 LLM 裁判判定为功能性记忆。
*   **文本指标严重低估风险**：仅依靠 BLEU 指标（阈值 0.75）只能检测到约 0.58% 的记忆样本，漏掉了大量逻辑层面的泄露。
*   **执行验证的挑战**：基于执行的检测（HyClone）最为严格，判定了 **0.28%** 的样本为完全一致的克隆，但受限于环境配置（如缺失库、类状态依赖），该比例是泄露风险的下限。
*   **无差别记忆**：模型对代码的记忆并不区分逻辑的敏感程度，普通逻辑和高价值逻辑均会被记忆。

#### 7. 优点
*   **视角新颖**：首次系统性地定义并量化了代码模型中“逻辑一致但文本不同”的记忆现象。
*   **实验设计精巧**：利用开源模型 Olmo-3 的透明性，构建了近乎完美的受控对比实验。
*   **工具链完整**：结合了静态分析（AST）、语义分析（LLM）和动态分析（执行），提供了全面的审计视角。

#### 8. 不足与局限
*   **执行覆盖率受限**：由于真实世界代码依赖复杂（如需要特定的类成员变量、外部数据库或库），大量样本无法通过执行测试进行验证（仅 15% 的候选样本成功执行）。
*   **LLM 裁判的主观性**：实验显示不同的 Prompt 会显著影响 LLM 裁判的评分结果，存在一定的随机性和偏差。
*   **语言单一性**：研究仅集中在 Python 语言，尚未验证在其他编程语言（如 C++、Java）或低频语言中的表现。
