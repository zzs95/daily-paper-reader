*   **核心问题**：现有的 Web 编程基准测试（如 HumanEval, SWE-Bench）主要关注算法正确性或文本驱动的生成，忽略了 Web 开发中至关重要的**视觉忠实度（Visual Fidelity）**、**交互质量**以及**多文件代码库级别的推理能力**。
*   **研究背景**：LLM 正在从被动助手演变为端到端的编程智能体，但缺乏一个能覆盖 Web 开发全生命周期（生成、编辑、修复）且支持多模态输入（文本、图像、视频）的统一评估框架。

### 2. 方法论：核心思想与关键技术
WebCompass 提出了一个多模态、全生命周期的评估体系，核心思想在于**“任务感知（Task-aware）”的评估范式**：
*   **任务分类**：涵盖 3 种模态（文本、图像、视频）和 3 种任务（生成、编辑、修复），细分为 7 个子类别。
*   **Agent-as-a-Judge（针对生成任务）**：
    *   引入 **Claude Code** 作为评估编排器，通过 **Model Context Protocol (MCP)** 控制真实浏览器。
    *   **流程**：自动生成测试清单 -> 浏览器交互 -> 动态合成 JavaScript 测试用例 -> 提取 DOM/截图证据 -> 评分。
    *   **创新点**：允许 Agent 根据生成的代码动态调整选择器（Selector），但保持行为断言（Assertion）不变，模拟人类的验收测试。
*   **LLM-as-a-Judge（针对编辑与修复任务）**：
    *   采用清单引导（Checklist-guided）模式，结合修改前后的截图、代码 Diff 和运行日志进行多维度评分。
*   **评分维度**：每个任务均从**可执行性（Executability）**、**功能性（Functional）**和**视觉质量（Visual）**三个轴线进行 0-10 分的评估，并使用**调和平均数**聚合，以惩罚表现不均衡的模型。

### 6. 主要结论与发现
*   **闭源模型领先**：Claude-4.5-Opus 和 Gemini-3-Pro 表现最强且最均衡，开源模型（如 Qwen3-VL-235B）仍有约 25 分的差距。
*   **视觉是瓶颈**：所有模型在“视觉质量”维度的得分普遍低于功能性得分，审美和精确布局是持久的挑战。
*   **任务难度差异**：编辑任务对开源模型极具挑战；修复任务中，语义缺陷（Semantic Error）比视觉缺陷更难修复。
*   **框架敏感性**：Vanilla（原生 JS）通常得分最高，Vue 框架由于单文件组件的复杂性，对模型最具挑战。
*   **思考模式的双刃剑**：Thinking 模式能提高代码结构正确性，但长链推理可能导致“注意力稀释”，反而遗漏部分功能需求。

### 7. 优点与亮点
*   **多模态全覆盖**：首次将视频输入（录屏）纳入 Web 编程评估，捕捉动态交互需求。
*   **Agent-as-a-Judge 范式**：利用 MCP 协议实现“智能体评测智能体”，解决了静态代码检查无法评估动态交互的难题。
*   **确定性修复任务**：通过“逆向注入缺陷”的方式构建修复任务，确保了评估的确定性和可重复性。

### 8. 不足与局限
*   **后端缺失**：仅关注前端（HTML/CSS/JS），未涉及数据库、API 等全栈开发能力。
*   **评估成本高昂**：依赖高阶模型（如 Claude-Opus）作为裁判，大规模推广的经济成本较高。
*   **动态性限制**：对于高度动态的场景（如 Web 游戏或复杂动画），目前的自动化交互评估仍难以完全替代人类的实时感官判断。
*   **污染风险**：作为静态基准，未来模型可能会在训练中包含这些数据，需定期更新以防数据污染。
